COUNTS

Goal: The count() function in Cypher is an aggregation tool used to quantify results returned by a query.

COUNTING ACROSS FEATURES

Goal: It operates in two distinct ways:

  • Key 01: count(*): Counts every row produced, including those containing null values.
  • Key 02: count(expression): Counts only non-null values for a specific property or variable.
  • Key 03: Summarizers (Aggregations): These are math operations like count(), sum(), or avg(). They collapse data into a single number.
  • Key 04: Organizers (Grouping Keys): These are any raw properties or labels (e.g., Category, Status, UserType).
  • Key 05: Cypher treats every Organizer as a "bucket label." If you list an Organizer, the engine creates a separate bucket for every unique value it finds.
  • Key 06: The Summarizers then perform their math inside each of those specific buckets.
COUNTING RELATIONSHIP TYPES

Use size() accesses the relationship store, where Neo4j keeps pre-calculated metadata on the node— Using MATCH requires traversal, where the engine visits every relationship, increasing cardinality.

One counts by doing work; the other just reads the saved total. To count relationships by type, you use a generic pattern without specifying labels.

MATCH ()-[r]->()

In the RETURN list, the type(r) function extracts the name of the relationship (the Organizer), while count(*) provides the math (the Summarizer). If you don’t use a Summarizer (like count()), Cypher doesn't know you want a total; it thinks you want a list. If you just RETURN type(r), the engine outputs the type for every single relationship it finds—meaning thousands of repetitive rows.

TIME COUNTS (DISTRIBUTION OF DATA)

Temporal boundaries: Identify the range of time covered by its entries.

  • By applying aggregation functions—specifically min() and max()—to a timestamp property on a set of nodes, you can isolate the earliest and latest data points in the graph.
  • To understand the distribution of data, you should group these nodes into discrete intervals, such as years or months.
  • To filter isolate a temporal moment, use a WHERE clause to filter by date.
  • Placing both year and month in the RETURN clause, Cypher uses them as grouping keys— This means the database organizes all matching nodes into "buckets" based on their shared year and month before performing the final tally.
FILTERING GRAPH PATTERNS

Cardinality is the total number of rows in the input stream— counting input size, not unique entities.

Existential Subqueries

Complex Filtering: You can use EXISTS { MATCH ... WHERE ... } to filter nodes based on a relationship that must also meet a specific property.

  • This lets you check for the existence of a qualified pattern without bringing those extra nodes into your main result set.
  • Relationship Absence (True Negation): Use NOT EXISTS { (u)-[:FRIEND]->() } to find "lonely" nodes.
  • This is the only way to effectively filter for the total absence of a relationship type.
  • While MATCH finds what is there, NOT EXISTS {} allows you to filter out nodes that satisfy a specific pattern of negating some "match" query.
  • Multi-Level Validation: You can nest patterns to validate a chain of existence (e.g. User posted a message that has a specific tag) while keeping the main query's cardinality focused solely on the User.
