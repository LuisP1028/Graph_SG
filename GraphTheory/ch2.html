<<Basic Graph Representations>>

ADJACENCY MATRIX:

The most basic graph representation is the mathematical data structure adjacency matrix. An adjacency matrix is a square matrix, where the matrix elements indicate whether pairs of nodes are connected or not in the graph. The adjancency matrix dimensions are equal to the number of nodes in the graph.

EDGE LIST

Simple data structure where each row represents a relationship of a given network. The value of the relationship weight is stored in a separate column. You could also store additional information about the relationship in the edge list.

Cypher Syntax (ASCII-Art)

To represent this via text, the Cypher language uses "ASCII-art" to make the code look like the graph it describes: • Nodes ( ): Parentheses represent the circle of a node. Inside, you define the Type (e.g., :Person) and Properties (e.g., {name: "Thomas"}). • Relationships -[]->: Brackets inside an arrow represent the link. Relationships must have a source and target; they cannot exist in a vacuum.

NATIVE GRAPH

A native graph database is typically a type of NoSQL database designed to store graph representations. Index free adjacency: Instead of an index, which would require O(1) pointers, keeping speed constant as the graph grows. each node has pointers to other nodes it has a relationship with, facilitating traversal across query nodes.

Resource Description Framework Database:

Structured as a triple is a data entity that consists of subject, predicate, and object. RDF is particularly suitable for creating ontologies that describe a specific domain of knowledge. Functioning like a basic sentence: Subject: The resource being described (an IRI or blank node). Predicate: The specific relationship or property (an IRI). Object: The value (Literal) or related entity (IRI/blank node). Nodes in an RDF graph: Internationalized resource identifier (IRI): Denotes any realworld entity or concept. An IRI is used to identify nodes in the graph in an unambiguous way. It is like an ID. Literal: Literals are values used to represent datatypes like strings, numbers, or dates. Blank node: Blank nodes are nodes without an identifier used in special RDF modeling scenarios.

Labeled-property graph database

Unlike simpler models, both nodes (the objects) and relationships (the connections) can store extra data as key-value pairs (properties) Labels categorize nodes (e.g., "Person" or "Product") Properties provide details (e.g., "Name: Alice"). This internal structure makes the graph highly flexible for modeling complex real-world data.

1. Monopartite Networks: Homogeneous Flows Definition: A graph where all nodes belong to a single category (e.g., $Node_A \to Node_A$). When to use: When you are modeling a peer-to-peer system where the "nature" of the entity doesn't change, only the direction of the influence. Logic: Use this for recursive importance (like PageRank). Influence flows through a single "type" of pipe, making it ideal for ranking, community detection, or pathfinding within a single domain (e.g., bank accounts, web pages, or social peers).

2. Bipartite (and K-Partite) Networks: Heterogeneous Mapping Definition: Edges can only exist between a node in one set and a node in the other. You cannot have an edge connecting two nodes within the same set. (e.g., $Node_A \to Node_B$). When to use: Use this to model ownership, consumption, or authorship. It separates the "actor" from the "object." Ontological Precision: By using specific labels (rather than generic ones), you "future-proof" the model against data collision when merging diverse datasets. It ensures that $Node_B$ and $Node_C$ don't get blurred into a single, unmanageable category.

3. Reified Interactions (Nodes as Actions) Definition: Turning a relationship into a standalone node (Reification). When to use: Use this when an interaction must be interactable. If you need to "like," "tag," or "comment" on an action that happened between two other nodes, that action must be a node. Logic: Since most graph models cannot point an edge at another edge, reifying the interaction allows you to attach metadata and incoming connections to the action itself without losing the history of the original event.

4. Hierarchy of Citation: Directed Provenance

Definition: Uses directed edges to track the origin of information. A Derivative Node (the copy or mention) has an outgoing relationship pointing to the Source Node (the original). When to use: Use this to preserve the Source of Truth. Logic: Outgoing (Derivative): The new/derived node points "back" to indicate where it came from. Incoming (Source): The original entity only receives links, marking it as the definitive target. This allows you to measure an object's reach or popularity simply by counting its "in-degree" (incoming links).

Data Storage: Separate/Within Node When choosing between storing data as node properties or separate nodes, the decision rests on your query goals and scalability. The Trade-off Node Properties: Storing a list of URLs directly on a node is simple but slow for discovery. To find nodes sharing a link, the database must scan every node's list property—essentially a costly search. Separate Nodes: Use Separate Nodes for shared data to leverage index-free adjacency over property scanning. Standardization Separate nodes work best when data is standardized. For example, "NYC" and "New York City" should point to the same node to ensure your graph remains unified and searchable.

Simplifying Complex Relationships

Relationship Specificity By using Specific Edges [Relationships] (e.g., PURCHASED, WROTE), you create Semantic Filters. During a traversal, the database engine ignores irrelevant edges entirely, only following the specific "type" required. Network Folding Simplifies complex relationships by converting indirect paths into direct connections. It is appropriate whenever you want to measure the strength of a relationship between two similar entities based on a shared "anchor."

Applying Bipartite Networks [Image of a bipartite graph]

Bipartite Projection (Folding) is your primary tool for distilling indirect associations into direct, measurable community clusters. Directed Boundaries: Apply this when an action starts in Set A and targets Set B, but entities within the same set do not interact directly in that specific context. Discovery via Projection: This is the core utility. By linking two nodes in Set A because they share a common neighbor in Set B, you project a new network. This reveals "co-occurrence" or hidden community clusters.
