CYPHER QUERY LANGUAGE & NEO4J MODULE_V1.0
Basic Query Language:

1. Nodes (Entities)
Nodes represent entities and are wrapped in parentheses ().

Syntax: (variable:Label {key: "value"})

Variable: A reference (e.g., p) used to access the node later in the statement.
Label: Preceded by a colon (e.g., :Person).

2. Relationships (Edges)
Relationships connect nodes and are wrapped in square brackets [] within arrows.

Syntax: -[variable:TYPE]->

Direction: Defined by --> or <--. Use -- to query without direction.
Type: Similar to labels, preceded by a colon (e.g., [:FRIEND]).

3. Properties
Metadata stored as key-value pairs inside curly brackets {}.
These can be assigned to both nodes and relationships to add context.

Primary Query Language Clauses MODULE_V1.0
Primary Query Language Clauses

MATCH: The most common clause;
it searches for patterns in the graph that match the structure you’ve defined.

Aerospace: MATCH (r:Rocket)-[:CARRIES]->(p:Payload)

WHERE: Adds constraints to a MATCH or OPTIONAL MATCH to filter results based on specific property values or logic.

Infrastructure: WHERE g.load > 0.85

CREATE / MERGE: CREATE adds new nodes or relationships;
MERGE ensures they exist (creating them only if they don't already).

Defense: CREATE (s)-[:COMMUNICATES_WITH]->(c:CommandCenter)

RETURN: Defines which data (nodes, relationships, or properties) should be output to the user.

Healthcare: RETURN d.specialty, count(p)

WITH: A powerful "pipe" that passes the results of one query part to the next, allowing for complex multi-step processing.

Defense: MATCH (u:Unit) WITH u WHERE u.fuel < 0.2 MATCH (u)-[:NEAREST]->(b:Base) RETURN b.name

SET: Property & Label Update
The SET clause is used to update properties on nodes and relationships, or to add new labels to existing nodes.
It is the primary way to "mutate" your data.

Infrastructure: MATCH (t:Transformer {id: 'TX-402'}) SET t.status = 'Maintenance Required'

REMOVE: Property & Label Deletion
REMOVE is the inverse of SET.
It is used to delete a specific property from an element or to strip a label from a node without deleting the node itself.

Healthcare: MATCH (p:Patient) REMOVE p.temporary_id

DELETE: Node & Relationship Removal
DELETE is used to remove nodes and relationships from the database.
Crucial Rule: In Neo4j, you cannot delete a node if it still has existing relationships attached to it (this maintains referential integrity).

Infrastructure: MATCH (s:Sensor)-[r:LINKED_TO]->(m:Monitor) DELETE r (Deletes only the connection).

DETACH DELETE: The "Force" Removal
DETACH DELETE is a shortcut that first deletes all relationships connected to a node and then deletes the node itself.
It is the standard way to "wipe" an entity from the graph entirely.

Aerospace: MATCH (s:Satellite {status: 'De-orbited'}) DETACH DELETE s
Importing CSV files to Cypher MODULE_V1.0
Importing CSV files to Cypher
Rules

1. Integrity: The "No Dupes” Rule
Integrity ensures every real-world object has exactly one node in your graph.
Without it, you might have two "Flight 101" nodes with different data.
Constraints act as a guardrail, blocking duplicates to keep your data "the single source of truth."

2. Performance: The "Fast Pass"
Performance is about speed. Without a constraint (and its automatic index), Neo4j must search every node to find one ID.
With it, the database uses a "map" to jump straight to the data, turning a slow, full-database crawl into a near-instant lookup.

CSV SPLITTING

Use different clauses for breaking apart the CSV, with columns being nodes, and relationships (edges) identified through different methods:

1. The "Wide Row" Method (Self-Contained)
The CSV contains the Source ID and Target ID on the same row.
First, you perform an initial pass to ensure all starting points exist;
this uses MERGE to create a node for every entry in your "Source" column, preventing duplicates.
Next, you run a second pass targeting the "Destination" column, performing the same MERGE operation to guarantee that every target entity is represented as a node in the database.
Finally, you execute the third pass, which acts as the connector.
In this step, you use MATCH to locate the existing source and target nodes defined in the first two passes and use MERGE to draw the relationship (the edge) between them.
This separated approach ensures data integrity and prevents the database from creating redundant nodes while it tries to build connections.
Use Case: A flight log: [Flight_ID, Origin_Airport, Destination_Airport].

2. The "Bridge Table" Method (Mapping)
The Bridge Table method is used when your nodes and your relationships live in separate files.
Step 1: The Entity Pass (Node Population)
You load the primary entity file (e.g., Airports.csv).
You use the MERGE clause on a unique property, specifically the IATA code (e.g., JFK, LAX).
The database creates a node for every airport. Descriptive metadata like "City Name" or "Number of Runways" are stored as properties within the node, but the IATA code serves as the primary unique identifier.
You are pre-populating the graph with these indexed "anchor points."
Step 2: The Mapping Pass (Instructional Input)
You switch to the Bridge CSV (e.g., Flight_Schedule.csv).
This file represents the topological logic of your graph. It contains foreign keys: an origin_code column and a destination_code column.
Cypher treats each row of this flight schedule as a set of instructions.
It reads the IATA codes from the CSV and prepares to use them as search parameters to locate the specific Airport nodes created in Step 1.
Step 3: The Assembly Pass (Relationship Execution)
This occurs in the same execution block as Step 2. Using the IATA codes retrieved from the schedule, Cypher performs a MATCH operation to locate the nodes in the database's memory.
It searches for the existing node where code = CSV.origin_code. It searches for the existing node where code = CSV.destination_code.
Once the pointers for these two specific nodes are identified, it executes a MERGE on the relationship: (origin)-[:FLIGHT_TO]->(destination).
Use Case: A hospital system where one file is Patients.csv and another is Doctor_Patient_Links.csv.

How Cypher "Identifies" the Edge
Regardless of the method, Cypher uses the MATCH clause to find the specific nodes you already imported and the MERGE clause to connect them
