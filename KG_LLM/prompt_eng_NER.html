2. NAMING OF ENTITY AND RELATION CLASSES

In the prompt, include lists of the entity and relation classes you’re most interested in. This helps clarify the task and provide more normalized outputs. The chosen terminology matters a lot! If you notice that an entity or relation class underperforms, try renaming it.

3. COMPLEX AND REPRESENTATIVE EXAMPLES

In complex tasks such as entity relation extraction (RE), it is useful to provide an example of input and expected output as part of the prompt.

  • Key 01: It can be a condensed version of the actual text you need to process, containing examples of important entities and relations.
  • Key 02: Include complex linguistic formulations that appear in the dataset, and focus on relations that the model struggles with.
CONFIGURATION AND STABILITY MODULE_V1.0
4. LLM CONFIGURATION

Experiment with different LLMs and choose the best option for you. Also test their parameters, especially temperature.

  • Key 01: The higher the temperature, the more varied and creative the output produced (and the greater the propensity for hallucinations).
  • Key 02: Lower temperatures yield more deterministic output.
// Temperature Guidelines Text Generation: Higher values Code Generation: Lower values (around 0.2) Entity/Relation Extraction: Temperature 0

For fact-focused tasks such as extracting entities and relations from a document, we use temperature 0.

5. TESTING PREDICTION STABILITY

Remember that LLMs are generative models, so by definition, they can have and do have “moods”: the same prompt can produce different results if run repeatedly on the same text. You can limit this behavior with careful prompt engineering (a non-ambiguous task formulation) and by setting very low values for temperature (as just discussed). If you choose a non-zero temperature, consider testing prediction stability by running the same prompt on the same test dataset multiple times and evaluating the overlap.

RIGOR AND EVALUATION MODULE_V1.0
4. UNIT-TESTING THE PROMPT

Goal: Prompt Unit-Testing applies software rigor to LLM engineering.

  • Key 01: Instead of manual "vibes" checks, you maintain a test suite of fixed inputs and expected outputs.
  • Key 02: By running this batch after every prompt iteration, you calculate a success rate and identify regressions, ensuring new updates don't break previously solved edge cases.
5. EYEBALLING A MINI-KG

Each time you reach a prompt milestone, we recommend deploying the prompt on a small sample of documents (a few dozen pages) and producing a KG. We find that a quick look at a mini-KG can help us discover opportunities for improvement.

6. EVALUATION

When you’re satisfied with the prompt, do a quantitative evaluation. calculate per-class precision, recall, and F1 score for entities and relations. use an LLM as a judge: instead of asking a human, give this task to another powerful LLM. It will make mistakes, but human-level performance is not 100% either
