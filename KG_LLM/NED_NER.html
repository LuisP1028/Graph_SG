NAMED ENTITY DISAMBIGUATION MODULE_V1.0
FOUNDATIONAL CONCEPTS

Goal: Disambiguate mentions by connecting them to structured representations in a specific domain Knowledge base.

  • NER: While Named Entity Recognition (NER) identifies that a word is a specific type (e.g., "Paris" is a Location), it doesn't specify which "Paris" it is—the city in France, the one in Texas, or Paris Hilton.
  • Logic: To filter this, NED uses candidate selection, ranking, and ontology integration.

NER: "Found a Person named 'Jordan'."

Candidate Gen: "Is it Michael Jordan (B-ball), Michael B. Jordan (Actor), or Jordan (Country)?"

Ranking: "The text mentions 'slam dunk,' so Michael Jordan (B-ball) moves to #1."

SciSpaCy Overview

1. Candidate Selection: scispacy utilizes a high-speed Approximate Nearest Neighbors (ANN) search based on character n-grams. It converts mentions into vectors of overlapping character sequences (e.g., "Zika" "zi", "ik", "ka") and compares them against the entire UMLS vocabulary.

2. Candidate Ranking: Once candidates are selected, the EntityLinker computes a similarity score for each. In scispacy, the similarity score is a TF-IDF weighted overlap of character n-grams. It then calculates the cosine similarity between these vectors. Essentially, it mathematically measures how much the "spelling pattern" of your text matches the "spelling patterns" of entities in the knowledge base.

3. Ontology Integration: This is the "hook" into the ontology. Scispacy moves beyond simple text matching and integrates the mention into a structured hierarchy. By connecting the term to an ontology, you can now broaden your search to that ontology's taxonomy.

ONTOLOGY CREATION WORKFLOW MODULE_V1.1
ONTOLOGY CREATION FROM DIFFERENT DATA SOURCES WORKFLOW

Phase 1: Ingestion & Digitization (Data Harvesting)

  • Extraction: Use OCR or PDF parsers to pull text from unstructured documents like shipping manifests, legal contracts, or news feeds.
  • Structuring: Map that text into a standardized format (JSON). Here, you maintain Data Provenance by tagging every snippet with its origin (e.g., source: "Port_of_LA_Log_2025") for auditing.

Phase 2: Disambiguation (Entity Linking)

  • Problem: Resolve the "same thing, different name" problem (e.g., "Apple Inc." vs "AAPL").
  • NER: An NLP model (like an LLM or spaCy) identifies "Apple" as an Organization.
  • Linking: Compare mention against an authoritative Knowledge Base (like Wikidata or a Corporate Master List) and assign a CUI Assignment unique ID (node fingerprint).

Phase 3: Loading & Integration (Graph Fusion)

  • Specialized Load: Import specialized domain files (e.g., Global Shipping Lanes and Regional Port Laws) to establish Logistics and Regulatory Logic.
  • Master Mapping: Apply Universal Industry Codes (e.g., HS Codes or UN/LOCODEs) to nodes so different data silos "speak" the same language.
  • Unification: Merge nodes with identical Universal IDs to create a single, Unified Ontology.
ADVANCED USE CASES MODULE_V1.2
ENABLING ADVANCED USE CASES OF YOUR ONTOLOGY

1. Structured Knowledge-Based Search: Formalized Navigation

In a unified ontology, search shifts from "looking for words" to "traversing predicates." The system navigates the formal path: Product (Electric Vehicle) -> Component (Battery) -> Material (Lithium) -> Port (Shanghai) -> Status (Congested).

KG-Based Interpretability & Discovery

  • Interpretability: Interprets the sentence by providing a logical justification for why terms appear together. (Cargo Ship)-[:DOCKS_AT]->(Deep-water Port).
  • Discovery: Reveals hidden context that text ignored by finding "bridge nodes." Example: Finding a biological link between "AIDS" and "Lupus" via the [:FINDING_SITE] in the Immune System.

3. Uncovering New Knowledge: Proactive Discovery Engine

This predictive layer identifies relationships by reversing the flow: using Text to repair and expand the Ontology. It identifies a Logical Gap where a high-frequency "Green Edge" (co-occurrence in live news) exists without a corresponding "Blue Edge" (formal ontological path) in the Knowledge Graph.

4. Graph-Based Filtering (Ontology Pruning)

Step 1: Garbage Collection (Weakly Connected Components)

Algorithm: Weakly Connected Components (WCC). Eliminates isolated "islands" typically caused by parsing errors or irrelevant text fragments.

Step 2: Contextual Scoping (Community Detection)

Algorithm: Louvain or Leiden. Optimizes modularity to group nodes into densely connected clusters (e.g., separating "Immunology" from "Hospital Billing").

Step 3: Hub Removal (Centrality Scoring)

Algorithm: Degree Centrality or Betweenness Centrality. Identifies and suppresses "Super-Nodes" (e.g., "Disease," "Location") that create the "Fan-Out Effect" and noise.

Step 4: Influence Filtering (PageRank)

Algorithm: PageRank. Deprioritize generic "background" knowledge. Filter out nodes with high Global PageRank but low Local Specificity.

Step 5: Constrained Pathfinding (The Discovery Query)

Algorithm: Shortest Path or Yen’s K-Shortest Paths. Finds the semantic link between two entities (e.g., "Zika" and "Rash") while excluding blacklisted hub nodes.
